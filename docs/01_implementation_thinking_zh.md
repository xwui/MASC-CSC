# MASC-CSC 实现思路

## 1. 项目的核心立意（SCI论文视角）

MASC-CSC 的核心立意指向了大语言模型（LLM）在拼写纠错中最致命的弱点：**过度纠正（Over-correction）**。
大模型拥有强大的语义推理，却缺乏对中文拼写错误产生机制（音近、形近）的感知。当它遇到一个微小的拼写错误时，往往会把它当做“开放式润色”，进而随意改写原句。

本项目提出：**让前端多模态小模型充当“报错传感器”，赋予 LLM 错误机制感知（Error Mechanism Awareness）能力。**
围绕这个核心立意，本项目致力于实现三大创新：
1.  **CMED (对比学习增强检测)**：强化前端查错的敏锐度。
2.  **ACG (自适应置信度门控)**：让前端模型预测“该不该问大模型”，节省开销的同时防误伤。
3.  **MCCD (多模态混淆集约束解码)**：利用前端的音形特征生成“安全字库”，从 Logit 层彻底封死 LLM 的发散式瞎改。

## 2. 为什么要这么设计

中文拼写纠错和一般文本生成不同，具有几个很强的任务约束：

- 大多数位置应当复制原字
- 错误通常是少量局部替换
- 尽量保持句长不变
- 不应进行风格润色或语义改写

而大语言模型天然更适合开放式生成，因此在 CSC 中容易出现：

- 过度纠错
- 改写句子
- 长度变化
- 修改本来正确但不常见的表达

另一方面，小模型尤其擅长：

- 局部字符判断
- 音近、形近错误建模
- 保守纠错

因此最合理的协作方式不是“LLM 主纠错”，而是：

- 小模型先给出高质量、受约束的候选
- LLM 再做有边界的选择

## 3. 整个系统的核心思想

MASC-CSC 采用四段式实现：

1. `Multimodal Frontend`
2. `Mechanism Inference`
3. `Constrained Candidate Generation`
4. `Selective Local Verification`

这四步分别对应下面的职责。

### 3.1 前端多模态模型

前端模型负责生成所有协同推理所需的基础信号：

- 每个位置的 logits
- 每个位置的 top-k 候选字
- 每个位置的检测分数
- 每个位置的不确定度

也就是说，前端模型不只是输出一个最终纠错句，而是输出一组可以供后续协同层使用的“中间证据”。

### 3.2 错误机制推断

在 CSC 中，很多错误具有机制性：

- 音近错误
- 形近错误
- 其他难以归类的错误

MASC-CSC 不直接把拼音和字形特征塞进 LLM，而是先把它们转化为“错误机制判断”。

这一步的意义是：

- 让系统知道“这个错为什么容易错”
- 让后续候选生成和 LLM 验证更有约束

### 3.3 候选句生成

候选生成器遵循几个硬约束：

- 只替换
- 不插入
- 不删除
- 原句必须保留
- 候选集要小

这样做的目的，是把原本开放的生成问题变成一个受控的选择问题。

### 3.4 风险路由与本地 LLM 验证

不是所有句子都需要 LLM。

因此系统先利用：

- 检测分数
- 熵
- margin
- 修改位置数

来判断一句话是否高风险。

只有高风险句子才会被送给本地 LLM verifier。

LLM 的职责是：

- 在候选中选择最优结果
- 判断是否应该保留原句

而不是自由生成新句子。

## 4. 当前实现与论文目标的巨大代差 (The Gap)

当前代码中，已经通过启发式规则搭通了管线（Pipeline），**但所有的核心创新点在代码中依然是空的或由规则代替**，距离真实的深度学习模型还有本质区别。

### 【当前已完成的管线骨架】
- 前端模型的 `predict_with_metadata()` (提供拼音/字形基础特征)
- 机制推断器的 **基于 pypinyin/余弦相似度的硬规则推断** (代替了 CMED 和机制分类头)
- 候选生成器 **按规则过滤**
- 风险路由器的 **四个写死硬阈值** (代替了可学习的 ACG 门控)
- 本地 LLM verifier 的 **单句选择题 Prompt** (代替了底层的 MCCD 编解码约束)

### 【面向 SCI 真正需要重构的核心】
1. **模型层改造 (CMED)**：给前方模型增加 Explicit Detection Head 与多分类 Mechanism Head，配合对比学习 Loss 端到端学习音形判别。
2. **训练层突破 (ACG)**：抛弃启发式算分，新增 `train_router.py`，把门控 α 变成一个能在验证集上端到端学出来的监督网络 / 强化策略网络。
3. **解码层重构 (MCCD)**：抛弃 ABCD 多选题 Prompt，编写自定义 `LogitsProcessor`，在 LLM next-token 预测时将非安全候选字 Logits 置为 `-inf`。
4. **验证层基准**：建设全自动 SIGHAN 批量测试脚本，单独切分“分题型（音近/形近）F1”与“误改率”指标。

## 5. 推荐的实现推进顺序

如果你现在要正式进入开发，最推荐的顺序是：

1. 先确认 `predict_with_metadata()` 输出的 top-k 和检测分数是否可信
2. 再调 `MechanismInferencer`
3. 再调 `MechanismAwareCandidateGenerator`
4. 再调 `RiskAwareRouter`
5. 最后再接 LLM verifier

原因很简单：

- 如果前端输出不稳定，后面所有模块都会一起坏
- 如果候选质量不够，LLM 再强也帮不上忙

## 6. 项目当前最大的风险点

这部分很重要，别人接手时必须提前知道。

### 风险 1：机制推断还是启发式

目前的 `mechanism.py` 主要基于：

- 拼音相等
- 图像余弦相似度

来判断音近或形近。

这对第一版原型足够，但还不够稳。

### 风险 2：候选生成仍可能过宽或过窄

如果阈值太松：

- 候选太多
- LLM 选择困难

如果阈值太严：

- 正确答案可能压根不在候选里

### 风险 3：LLM verifier 现在是单句演示级入口

目前 `run_masc_csc.py` 适合单句验证，不适合直接大规模实验。

### 风险 4：训练与协同推理仍是两层逻辑

当前前端训练已经有入口，但协同层尚未形成统一实验脚本。

## 7. 一句话总结

MASC-CSC 的真正实现主线是：

> 先让前端小模型输出足够可靠的字符级证据，再用机制约束收缩候选空间，最后让本地 LLM 只在高风险样本上做有限选择。
